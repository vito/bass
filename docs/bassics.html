bassics

The following is a list of the built-in types and common terms you will run into when writing Bass programs.

symbol
𝄢

A name, typically bound to a value in a scope.

𝄢

A symbol form evaluates by fetching its binding in the current scope.

symbol?
result
(wrap <builtin op: (symbol? val)>)
𝄢

Symbols may be constructed using a keyword form, analogous to cons forms which construct pairs.

:symbol?
result
symbol?
𝄢

Symbols are sometimes used as constant values signalling certain conditions, such as :end to indicate the end of a stream.

(def vals (stream 1 2))

[(next vals :end)
 (next vals :end)
 (next vals :end)]
result
(
  1. 1
  2. 2
  3. end
)
𝄢

Symbols may be chained together with keyword notation to traverse scopes.

(def foo {:a {:b 42}})
result
foo 
foo:a:b
result
42
𝄢

Symbols are applicative combiners; when called with a scope, it fetches its binding from the scope.

A default value may be provided as a second argument.

(def foo 123)

[(:b {:a 1 :b 2 :c 3})
 (:foo (get-current-scope))
 (:b {:a 1} 42)
]
result
(
  1. 2
  2. 123
  3. 42
)
keyword
𝄢

A symbol prefixed with a : is called a keyword. It is used to construct the symbol itself rather than fetch its binding.

:im-a-symbol!
result
im-a-symbol!
𝄢

Keywords go hand-in-hand with cons, which is used to construct pairs.

[:+ 1 2 3]
result
(
  1. +
  2. 1
  3. 2
  4. 3
)
𝄢

Note: keywords evaluate to a symbol - they are not a distinct value.

scope
𝄢

A set of symbols bound to values, with a list of parent scopes to query (depth-first) when a local binding is not found.

𝄢

All code evaluates in a current scope, which is passed to operatives when they are called.

(defop here _ scope scope)

(let [local-binding "hello"]
  (here))
result
{
local-binding "hello"
{
here (op _ scope scope)
{
*args*
(
)
*dir* .
*stdin* <source: >
*stdout* <sink: >
<scope: ground>
}
}
}
𝄢

Scopes can also be constructed using {bind} notation, allowing them to be used like a hash or map:

(eval [str :uri "@" :branch]
  {:uri "https://github.com/vito/bass"
   :branch "main"})
result
"https://github.com/vito/bass@main"
𝄢

Parent scopes may be provided by listing them anywhere in between the braces. For example, here's a scope-based alternative to (let):

(defop with [child & body] parent
  (eval [do & body] {(eval child parent) parent}))

(with {:a 1 :b 2}
  (+ a b))
result
3
𝄢

Comments within the braces are recorded into the child scope, enabling their use for lightweight schema docs:

(eval [doc]
  {; hello world!
   :foo "sup"

   ; goodbye world!
   :bar "later"})
stderr 
--------------------------------------------------
foo string?
hello world!
--------------------------------------------------
bar string?
goodbye world!
result
null
𝄢

Scopes encode to JSON by converting hyphens to underscores in their bindings. This may be slightly surprising but overall it helps with interoperability while keeping Bass code looking Lispy.

(dump {:foo-bar "hello!"})
stderr 
{
  "foo_bar": "hello!"
}
result
{
foo-bar "hello!"
}
number
𝄢

An integer value. Floating point values are not supported.

(* 6 7)
result
42
string
𝄢

A UTF-8 immutable string value.

path
𝄢

A location of a file or directory within a filesystem.

𝄢

Bass distinguishes between file and directory paths by requiring a trailing slash (/) for directories.

(def file ./some-file)
(def dir ./some-dir/)
[file dir]
result
(
  1. ./some-file
  2. ./some-dir/
)
𝄢

Directory paths can be extended to form longer paths:

dir/file-in-dir
result
./some-dir/file-in-dir
𝄢

Extending a file path, however, will result in an error.

file/bam
stderr 
error! call trace (oldest first):
  1. (docs):1	file/bam
cannot extend path ./some-file (bass.FilePath) with bam (bass.FilePath)
null
𝄢

Everyone's old friend. Used to represent the absense of value where one might typically be expected.

null
result
null
𝄢

Note: null is a distinct type from an empty list. The only (null?) value is null, and the only (empty?) value is [].

(map null? [[] (list) null false])
result
(
  1. false
  2. false
  3. true
  4. false
)
boolean
𝄢

true or false, but sometimes null.

𝄢

Boolean values are pretty straightforward - the only catch is that null also counts as false when given to (if) or (not).

𝄢

Otherwise, all values - including "empty" ones - are truthy.

[(if true :truthy :falsy)
 (if false :truthy :falsy)
 (if null :truthy :falsy)
 (if [] :truthy :falsy)
 (if "" :truthy :falsy)
 (if _ :truthy :falsy)]
result
(
  1. truthy
  2. falsy
  3. falsy
  4. truthy
  5. truthy
  6. truthy
)
list
𝄢

A pair or cons whose second element is an empty list or a list.

𝄢

A pair form evaluates by combining its first value its second value - meaning the first value must be a combiner.

(list 1 2 3)
result
(
  1. 1
  2. 2
  3. 3
)
𝄢

A cons form evaluates like cons: it constructs a pair by evaluating each of its values.

[1 2 3]
result
(
  1. 1
  2. 2
  3. 3
)
𝄢

Both pair and cons may have a & symbol which provides a value for the rest of the list.

(def values [1 2 3])
result
values 
(+ & values)
result
6 
[-1 0 & values]
result
(
  1. -1
  2. 0
  3. 1
  4. 2
  5. 3
)
empty list
𝄢

An empty list is represented by () or [], which are both the same constant value.

(= [] ())
result
true
pair
𝄢

A list of forms wrapped in (parentheses), or constructed via cons or cons notation.

(= (cons 1 (cons 2 [])) (list 1 2))
result
true
𝄢

A & may be used to denote the second value instead of terminating with an empty list.

(= (cons 1 2) (list 1 & 2))
result
true
cons
𝄢

A list of forms wrapped in [square brackets].

(= (cons 1 (cons 2 [])) [1 2])
result
true
𝄢

A & may be used to denote the second value instead of terminating with an empty list.

(= (cons 1 2) [1 & 2])
result
true
ignore
𝄢

_ (underscore) is a special constant value used to discard a value when binding values in a scope.

(def (a & _) [1 2 3])

a ; the only binding in the current scope
result
1
𝄢

_ is also used when null is just not enough to express how absent a value is - for example, to record overarching commentary within a module.

; Hey ma, I'm a technical writer!
_
result
_
workload
𝄢

A command to be run with inputs provided on standard interfaces (env, stdin, args, etc.).

𝄢

Workloads are constructed by applying a path locating the command to run. The arguments to a path combiner provide values to be passed to the workload on stdin.

𝄢

Throughout this documentation, workloads will be rendered as space invaders. Under the hood, workloads are just scopes with certain bindings - you can click them to reveal their inner structure.

(.cat 42)
result
{
path
{
command "cat"
}
response
{
stdout true
}
stdin
(
  1. 42
)
}
𝄢

To specify values on argv instead, use (with-args) - or use the $ operative for syntactic sugar.

(def val "hello")

[(-> (.echo) (with-args "hi" val))
 ($ .echo hi $val)]
result

    (
    1. "hi"
    2. "hello"
    )
    path
    {
    command "echo"
    }
    response
    {
    stdout true
    }
    }
  2. {
    args
    (
    1. "hi"
    2. "hello"
    )
    path
    {
    command "echo"
    }
    response
    {
    stdout true
    }
    }
)
𝄢

Workloads are constructed by combining a series of builder functions provided by the stdlib.

The -> operative comes in especially handy for building bespoke workloads without wanting to claw your eyes out.

(-> (.cat 42) (in-image "alpine"))
result
{
image
{
repository "alpine"
}
path
{
command "cat"
}
platform
{
os "linux"
}
response
{
stdout true
}
stdin
(
  1. 42
)
}
𝄢

To run a workload, pass it to (run); it will return a source from which you can read values using (next).

Putting it all together into one big -> pipeline, you get:

(-> (.cat 42) (in-image "alpine") run next)
result
42
runtime
𝄢

An internal component used for running workloads, configured by the user and associated to a platform.

𝄢

Currently, only Docker is supported - Kubernetes support may come in the future.

platform
𝄢

A scope containing arbitrary bindings used to select the appropriate configured runtime for a given workload.

workload path
𝄢

A path paired with the workload that created (or creates) it.

(path (.run) ./some-file)
result
{
path
{
command "run"
}
response
{
stdout true
}
}
𝄢

Workload paths can be constructed and passed around to other workloads as first-class values. The runtime will handle the boring mechanical work of mounting it into the container in the right place.

(def touchi-artist
  (-> ($ .touch ./artist)
      (in-image "alpine")
      (path ./artist)))

(-> ($ .ls $touchi-artist)
    (response-from :exit)
    (in-image "alpine")
    run
    next)
result
0
𝄢

Workload paths represent reproducible artifacts - when passed from one workload into another, any paths created by the child workload will be inherently downstream of all of its inputs. The workload path may be exported to JSON format and used to re-create the artifact from scratch with bass -e < wlp.json.

pipe
𝄢

A pipe is a source with a corresponding sink.

source
𝄢

A stream of values which may be read with (next).

𝄢

All scripts can read values from the *stdin* source, which reads JSON encoded values from stdin.

A source may be constructed from a sequence of values by calling (stream).

(def vals (stream 1 2 3))
[(next vals)
 (next vals)
 (next vals)]
result
(
  1. 1
  2. 2
  3. 3
)
𝄢

A source is also returned by (run) to pass along values emitted by the workload.

When (next) hits the end of the stream, an error will be raised. A default value may be supplied as the second argument to prevent erroring.

sink
𝄢

A destination for values which may be sent with (emit).

𝄢

All scripts can emit values to the *stdout* sink, which encodes values as JSON to stdout.

(emit "hello!" *stdout*)
(emit 42 *stdout*)
stdout
"hello!"
42
result
null
combiner
𝄢

A value which can be applied to another value (typically a list of arguments) to perform some computation and return another value.

operative
𝄢

A combiner which receives the applied arguments, unevaluated, along with the dynamic scope of the caller.

𝄢

Operatives are defined with the (defop) operative or constructed with (op).

(defop quote-with-scope args scope
  [args scope])

(quote-with-scope a b c)
result
(
  1. (
    1. a
    2. b
    3. c
    )
  2. {
    quote-with-scope (op args scope [args scope])
    {
    *args*
    (
    )
    *dir* .
    *stdin* <source: >
    *stdout* <sink: >
    <scope: ground>
    }
    }
)
applicative
𝄢

A combiner which wraps an underlying operative and evaluates its arguments before passing them along to it.

𝄢

Applicatives, typically called functions, are defined with the (defn) operative or constructed with (fn).

(defn inc [x]
  (+ x 1))

(inc 41)
result
42
